# Comprehensive Codebase Analysis

## Objective
Transform source code into a complete, navigable documentation ecosystem that captures program structure, behavior, business logic, and architectural patterns with sufficient detail for reimplementation, maintenance, and comprehensive understanding. End with a final recommendation of which AWS-managed transformation(s) are recommended for the repository, if any, based on the analysis.

## Summary
This transformation performs deep static analysis of codebases to generate hierarchical, cross-referenced documentation covering all aspects of the system. It combines behavioral analysis, architectural documentation, and business intelligence extraction to create a comprehensive knowledge base organized for maximum usability and navigation. The transformation places special emphasis on technical debt analysis, providing prominent, actionable insights on outdated components, and maintenance concerns at the root level. This transformation operates strictly on static code analysis and NEVER attempts to build or execute the code. The transformation concludes with a final recommendation of which AWS-managed transformation(s) are recommended for the repository, if any, based on the analysis. If none apply, then the transformation still writes the recommended next steps to address technical debt in the various technical debt related files.

## Entry Criteria
1. Source code repository with readable files (compilation not required)
2. Partial or complete project structure accessible
3. Basic file system permissions for analysis
4. Sufficient storage for comprehensive documentation output
5. No external dependencies required for analysis
6. NO compilation, building, or execution of code required or permitted during this transformation
7. NO running of tests within the codebase is permitted during this transformation

## IMPORTANT: Exclusions
The following directories should be excluded during analysis to improve efficiency and focus on actual application code:
1. Package management directories:
   - node_modules, bower_components, jspm_packages, packages
   - .npm, .nuget, .gradle, .m2, .ivy2
   - venv, virtualenv, env, .env, .venv, .virtualenv
   - pip-wheel-metadata
2. Build output directories:
   - dist, build, target, out, output, publish, releases, release
   - bin, obj, Debug, Release
   - .output, .build, .dist
   - compiled, bundle, bundles
3. Generated code and documentation:
   - generated, auto-generated, autogenerated, gen, .gen
   - docs, documentation, javadoc, apidoc, swagger, openapi
4. Version control and tooling:
   - .git, .svn, .hg, .bzr
   - .github, .gitlab, .circleci
   - .idea, .vscode, .vs
   - .gradle, .m2, .npm, .yarn

   - Only at the very end, after all validation passes, COPY (duplicate) the completed ATXDocumentation/ folder to ~/.aws/atx/custom/{conversationId}/

         find . -type f \( -name '*.<extension>' -o -name '*.<extension>' \) ! -path '*/node_modules/*' ! -path '*/bower_components/*' ! -path '*/jspm_packages/*' ! -path '*/packages/*' ! -path '*/.npm/*' ! -path '*/.nuget/*' ! -path '*/.gradle/*' ! -path '*/.m2/*' ! -path '*/.ivy2/*' ! -path '*/venv/*' ! -path '*/virtualenv/*' ! -path '*/env/*' ! -path '*/.env/*' ! -path '*/.venv/*' ! -path '*/.virtualenv/*' ! -path '*/pip-wheel-metadata/*' ! -path '*/dist/*' ! -path '*/build/*' ! -path '*/target/*' ! -path '*/out/*' ! -path '*/output/*' ! -path '*/publish/*' ! -path '*/releases/*' ! -path '*/release/*' ! -path '*/bin/*' ! -path '*/obj/*' ! -path '*/Debug/*' ! -path '*/Release/*' ! -path '*/.output/*' ! -path '*/.build/*' ! -path '*/.dist/*' ! -path '*/compiled/*' ! -path '*/bundle/*' ! -path '*/bundles/*' ! -path '*/generated/*' ! -path '*/auto-generated/*' ! -path '*/autogenerated/*' ! -path '*/gen/*' ! -path '*/.gen/*' ! -path '*/docs/*' ! -path '*/documentation/*' ! -path '*/javadoc/*' ! -path '*/apidoc/*' ! -path '*/swagger/*' ! -path '*/openapi/*' ! -path '*/.git/*' ! -path '*/.svn/*' ! -path '*/.hg/*' ! -path '*/.bzr/*' ! -path '*/.github/*' ! -path '*/.gitlab/*' ! -path '*/.circleci/*' ! -path '*/.idea/*' ! -path '*/.vscode/*' ! -path '*/.vs/*' -exec cat {} + 2>/dev/null | wc -l
         ```
   2. If the codebase contains **more than 1,000,000 lines of code**, it is considered a **large codebase**
   3. If a large codebase is detected, prepend a disclaimer to every file in ATXDocumentation/ (USE THE MESSAGE EXACTLY AS WRITTEN):
      ```bash

Some sections may contain higher-level summaries rather than exhaustive detail. ### 2. Comprehensive Code Analysis
1. **Project discovery and inventory**
   - Read existing ATXDocumentation/ files if they exist to get context
   - Scan all source files and identify programming languages, frameworks, and technologies
   - Extract project metadata, configuration, and directory structure

2. **Extract program structure**
   - Document classes, interfaces, functions, and modules
   - Map inheritance hierarchies and composition relationships
   - Extract method signatures, parameters, and return types
   - Identify design patterns and architectural styles

3. **Analyze dependencies and relationships**
   - Map internal component dependencies (module-to-module, class-to-class relationships)
   - Identify ALL external library usage with exact versions
   - Create dependency graphs with criticality analysis
   - Document data flow between components
   - Analyze threading and concurrency patterns
   - Document build tool dependencies (Maven plugins, Gradle plugins, npm scripts)
   - Identify transitive dependencies where relevant
   - **Identify technical debt and outdated components**

4. **Extract behavioral information - FOCUS ON APPLICATION CODE ONLY**
   - **CRITICAL: Document ONLY workflows that represent core application business logic**
   - **EXCLUDE build/test/lint/publish workflows from workflow documentation**
   - Document control flow and execution paths for business operations
   - Map conditional logic and branching patterns in application code
   - Extract business rules and validation logic
   - Identify state machines and lifecycle patterns
   - Document exception handling and error recovery

### 3. Technical Debt Analysis and Documentation
1. **Generate root-level technical debt summary**
   - Create `technical-debt-report.md` at the root level for maximum visibility
   - **CRITICAL: At the TOP of technical-debt-report.md, include the AWS Transformation Recommendation section using EXACTLY this format:**
   
   **For a single recommended transformation:**
   ```markdown
   ## 🎯 AWS Transformation Recommendation
   
   ### **RECOMMENDED TRANSFORMATIONS: AWS/<transformation-name>**
   
   [Brief 2-3 sentence justification explaining why this transformation is recommended based on the codebase analysis]
   ```
   
   **For multiple recommended transformations:**
   ```markdown
   ## 🎯 AWS Transformation Recommendation
   
   ### **RECOMMENDED TRANSFORMATIONS: AWS/<transformation-name-1>, AWS/<transformation-name-2>**
   
   [Brief 2-3 sentence justification explaining why these transformations are recommended based on the codebase analysis]
   ```
   
   **If no AWS-managed transformation applies:**
   ```markdown
   ## 🎯 AWS Transformation Recommendation
   
   ### **RECOMMENDED TRANSFORMATIONS: None**
   
   [Brief 2-3 sentence explanation of why no AWS-managed transformation applies and what the recommended next steps are]
   ```
   
   - The transformation name MUST exactly match one from the AWS-managed transformations returned by the `list_available_transformations_from_registry` tool
   - This recommendation section MUST be the first section after any title/header in technical-debt-report.md
   - Do NOT proceed with applying any of these transformations - only recommend them
   - Include executive summary of technical debt findings
   - Provide prioritized list of critical issues with severity ratings (High/Medium/Low)
   - **CRITICAL: Always prioritize findings in this fixed order: (1) EOL/deprecated/outdated runtimes and frameworks, (2) outdated dependencies, (3) code quality and architectural issues**
   - Include navigation links to detailed technical debt sections
   - Add visual indicators for severity and impact

2. **Create dedicated technical debt section**
   - Organize detailed findings in the `technical-debt/` directory:
     - `summary.md`: Overview of all technical debt findings
     - `outdated-components.md`: Detailed analysis of obsolete components
     - `maintenance-burden.md`: Areas requiring significant maintenance attention
     - `remediation-plan.md`: Prioritized action items

3. **Perform comprehensive technical debt identification**
   
   **3.1 Outdated Components Analysis**
   - Analyze language/runtime versions and EOL status
   - Identify deprecated frameworks and libraries with exact versions
   - Document ALL outdated dependencies (not just a subset)
   - Include build tool versions (Gradle, Maven, npm, etc.)
   - Document deprecated algorithms or cryptographic methods
   - Assess coding conventions and standards compliance
   - Prioritize issues by risk level using qualitative terms (High/Medium/Low)
      - High — Outdated/EOL/deprecated runtimes and frameworks
      - Medium — Outdated runtime/production dependencies (libraries used at runtime)
      - Low — Outdated developer/build dependencies (test frameworks, linters, formatters, build tools, type definitions, dev-only tooling)
   
   **3.2 Additional Technical Debt**
   - Detect complex or unmaintainable code patterns
   - Detect performance bottlenecks and architectural issues

4. **Create actionable remediation recommendations**
   - Provide specific upgrade paths for outdated components
   - Suggest architectural improvements with concrete examples
   - Prioritize issues by risk level and criticality using qualitative terms (High/Medium/Low)
      - High - EOL/deprecated/outdated runtimes and frameworks
      - Medium - Outdated dependencies
      - Low - Code quality and architectural issues
   - Document specific requirements for addressing each debt item
   - Reference industry best practices and migration patterns
